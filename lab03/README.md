### 1. Описание проблемы предметной области

В системах биржевой торговли обработка входящих событий (рыночных тиков или заявок пользователя) является многоэтапным и динамическим процессом. Основные сложности включают:
*   **Сложная валидация:** Перед исполнением сделки необходимо проверить баланс, наличие активов, корректность рыночных цен и соответствие правилам управления рисками.
*   **Связанность кода:** Использование длинных цепочек `if-else` или вложенных условий делает код трудночитаемым и жестким. Изменение одного правила (например, добавление комиссии) требует вмешательства в основную логику исполнения.
*   **Разные режимы обработки:** Фоновое обновление котировок требует одного набора действий (пересчет прибыли, рендер), а пользовательский ордер — другого (валидация денег, проверка лимитов, расчет налогов).

### 2. Решение: Применение паттерна Chain of Responsibility

Для решения этих проблем в проекте реализован паттерн **Цепочка обязанностей (Chain of Responsibility)** в его современном виде — **Pipeline (Конвейер)**.

#### Архитектурные особенности:
*   **Объект контекста (`TradeContext`):** Все данные о событии (тикер, цена, состояние портфеля, ошибки сети) упаковываются в единый изменяемый объект, который передается по цепочке.
*   **Плоский конвейер (`Pipeline`):** Вместо классической вложенности обработчиков используется плоский список объектов `BaseHandler`. Это позволяет динамически собирать разные цепочки для разных задач.
*   **Механизм прерывания:** Каждое звено может установить флаг `stop_chain`. Если, например, `RiskManager` обнаруживает превышение лимитов, он останавливает дальнейшую обработку, предотвращая исполнение сделки.

#### Звенья цепи (Handlers):
1.  **`PriceSanitizer`**: Очищает и проверяет корректность данных от API Мосбиржи, рассчитывает цены покупки/продажи с учетом комиссии.
2.  **`AuthValidator`**: Проверяет достаточность средств на виртуальном балансе или наличие акций в портфеле.
3.  **`RiskManager`**: Реализует логику риск-менеджмента (запрет на покупку актива, если его доля в портфеле превысит 50%).
4.  **`ProfitLossCalculator`**: Пересчитывает текущую нереализованную прибыль (P&L) для всех открытых позиций.
5.  **`TerminalRenderer`**: Отвечает за визуализацию — отрисовку Dashbord-панели в консоли.

### 4. Вывод

Применение паттерна Chain of Responsibility позволило создать **модульную и расширяемую архитектуру** торгового симулятора. 

**Ключевые преимущества для проекта:**
*   **Чистота кода:** Логика проверки рисков полностью отделена от логики взаимодействия с API или визуализации.
*   **Гибкость:** Программа использует три разные конфигурации цепочек (`FULL`, `VALIDATION`, `PRESENTATION`), переиспользуя одни и те же классы обработчиков.
*   **Надежность:** Каждое звено изолированно обрабатывает свои исключения (например, сетевые ошибки в Sanitizer), не ломая логику работы всей системы.
